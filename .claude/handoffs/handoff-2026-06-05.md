## Next task: #52 (security — gate `/api/admin/*`)

Issue: https://github.com/bennetstrauch/tm-muenchen/issues/52 (full context + acceptance criteria there).

**The problem in one line:** `proxy.ts` gates `/admin` *pages* behind the session cookie but passes all `/api` through ungated. Of 22 files under `src/app/api/admin/`, only `email-send` checks auth (magic-link `x-admin-token`/`x-admin-token-event` headers); `login` is the login. The rest read/mutate data with no auth. Confirmed live: `GET`/`PUT /api/admin/einstellungen` worked with no cookie.

**Why it needs care (do this analysis first):** the fix must require a session cookie **OR** a valid magic-link token, or it will break Leiter (teacher) scoped access. Before editing the proxy, **map which `/api/admin/*` routes the magic-link client calls** — trace the fetches in `src/app/admin/admin-client.tsx` (~1700 lines; it receives `tokenHeader`/`tokenEventId` props in magic-link mode) and see which send `x-admin-token`. `email-send` is the known example of the token-header pattern. Decide per-route whether magic-link scope suffices or full-admin is required.

**Building blocks already in place:**
- `verifySessionToken(token, expectedTenant)` in `src/lib/admin-session.ts` — use this for the cookie check (same call the proxy already uses for `/admin` pages).
- `verifyToken(token, eventId)` in `src/lib/admin-token.ts` — the magic-link check.
- The proxy already resolves `slug` (tenant) before the auth branch, so `verifySessionToken(cookie, slug)` is the natural gate.

**Testing note:** the proxy has no unit tests (hard to test with NextRequest + next-intl) — pattern is to test the verify helpers (done) and smoke-test the proxy at runtime. For #52, add at least one regression test proving an ungated route rejects an unauthenticated request, and runtime-verify a magic-link flow still works.

## Auth design reference (implemented this session, for context)

- ADR 0006 = bcrypt password per tenant + signed session cookie. Honored.
- `src/lib/signed-token.ts` — shared HMAC `sign`/`verify` core (the only token crypto). Both `admin-token.ts` (magic links) and `admin-session.ts` (sessions) are thin wrappers on it.
- Session cookie = signed `{tenant, exp}` token, verified in `proxy.ts` and `admin/page.tsx` against the request's tenant. `ADMIN_USER`/`ADMIN_PASS` env vars are gone.
- `vitest.config.ts` added this session (the `@/` alias — needed to test app-code, not just `lib/`).
